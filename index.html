Capabilities

The full agent toolkit.

Bingwa Flash is designed around a real transaction lifecycle: detect a request, choose the right fulfilment path, run it once, verify it, and show clearly when agent action is needed.

01 / TRANSACTIONS

Payment-triggered automation

Relevant M-Pesa and configured SMS events can create transactions that progress through pending, executing, verified, completed, scheduled, skipped or failed states.

02 / MAPPING

USSD mappings and tracer

Build simple or advanced menu flows, map offers to SIMs, and use tracer support when carrier menu positions move unexpectedly.

03 / RECOVERY

Retry and alternative USSD

Retriable failures can follow configured retry rules or a selected alternative USSD path, without creating a second execution path.

04 / SAFETY

Verifier and held requests

Verifier messages can confirm completed fulfilment. Requests held by accessibility, automation, tokens or airtime can surface for agent confirmation when ready again.

05 / BALANCES

Airtime and token safeguards

Track airtime usage on completed work, manage token access, and optionally use emergency airtime purchase for an enabled unit-token offer when capacity reaches zero.

06 / WEB ORDERS

Public order pages

Agents can expose selected offers through their public profile, receive paid web requests and route them into the same controlled fulfilment workflow.

07 / CLIENTS

Client tools and lookup

Manage client records, website registration, contact saving, blacklist actions and masked-number resolution where supported by configured messages.

08 / RELAY

Relay and portal workflows

Use enabled sender and receiver rules for relay fulfilment, plus portal controls for authorised multi-device workflows when explicitly switched on.

09 / REPORTING

Alerts and performance

Review transaction history, failures, airtime usage, commission, offer breakdowns and alerts including emergency token purchase events.

Workflow

One request. One trail.

Whether a request starts on the phone, from a web order, or through an enabled relay, the agent can follow its outcome instead of guessing what happened.

STEP 01

Detect

A payment or approved request is received and matched.

STEP 02

Check

Offer, SIM, balance, tokens and automation readiness are evaluated.

STEP 03

Execute

The selected USSD mapping runs through the configured path.

STEP 04

Verify

Response or verifier confirmation finalises the outcome.

STEP 05

Report

History, airtime, commission and alerts remain visible.

Agent control

Automation that stays accountable.

Safaricom, M-Pesa, network prompts and Android background restrictions remain outside the app's control. Bingwa Flash gives agents tools to configure, inspect and recover work safely.

Setup essentials

Configure before going live

Reliable execution begins with the right phone state and verified mappings.

  • Correct SIM and USSD offer mappings
  • Accessibility, SMS and phone permissions enabled
  • Sufficient airtime or token access for the offer
  • Retry, verifier and reply rules reviewed
When blocked

Recover with a clear decision

When readiness returns after accessibility, automation, token or airtime interruptions, pending work can be reviewed rather than blindly executed.

  • Inspect failed and held transaction reasons
  • Choose queue, skip, ignore or retry where offered
  • Use fallback only for configured retriable failures
  • Keep a visible audit trail of final outcomes
